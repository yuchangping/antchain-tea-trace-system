# 茶叶溯源系统项目进度总表

## 1. 文档定位

这份文档不是普通记录表，而是项目总控驾驶舱。

它负责解决 5 件事：

1. 记录当前项目到底推进到哪一步
2. 固定当前应该先做什么、后做什么
3. 让每个重大改动都能走完“开发、测试、确认、文档、Git”闭环
4. 给每个阶段设置里程碑、出口条件和验证证据
5. 保证任何关键改动都有可回退的 Git 历史

使用顺序固定为：

1. 先读 `AGENTS.md`
2. 再看本表
3. 再去看 `茶叶溯源系统分阶段工作流程.md`
4. 最后进入代码或专题文档

---

## 2. 使用规则

### 2.1 更新时机

每次推进都必须更新本表，不能只在阶段结束时补记。

固定时机如下：

1. 开始一个新阶段前，更新“当前项目快照”和“当前窗口任务”
2. 开始一个重大改动前，先补“任务拆分、出口条件、Git 保护点”
3. 开发完成后，补测试和确认结果
4. 文档同步后，才允许把状态改成 `已完成`
5. 发现联调问题时，立即转为 `需修复` 或 `阻塞`

### 2.2 闭环门禁

每项重大改动必须走完下面顺序：

1. `规划`
2. `Git 保护`
3. `开发`
4. `测试`
5. `确认`
6. `文档`
7. `Git 收口`

缺任何一步，都不能记为完成。

### 2.3 文档联动

本表更新后，通常还要联动下面文档：

- 阶段规则变化：更新 `茶叶溯源系统分阶段工作流程.md`
- 项目入口、执行顺序、Git 规则变化：更新 `AGENTS.md`
- 运行方式或目录变化：更新 `tea-trace-system/README.md`
- 架构/API/数据库/演示变化：更新 `tea-trace-system/docs/*.md`

## 3. 状态字典

本项目统一使用下面这些状态：

| 状态 | 含义 | 是否可直接进入下一阶段 |
| --- | --- | --- |
| `未开始` | 还未规划或排期 | 否 |
| `规划中` | 已开始拆任务，但未进入开发 | 否 |
| `开发中` | 正在实现功能或文档 | 否 |
| `开发完成待测试` | 代码或文档主体已完成，等待验证 | 否 |
| `测试通过待确认` | 已通过测试，等待业务确认或演示确认 | 否 |
| `已确认待文档` | 功能成立，但文档尚未同步 | 否 |
| `待联调` | 单模块通过，等待跨模块联动验证 | 否 |
| `已完成` | 闭环完成，可作为稳定基础 | 是 |
| `需修复` | 已发现问题，需要回头修复 | 否 |
| `阻塞` | 存在卡点，当前不能继续推进 | 否 |

## 11. 使用这张表的最终原则

以后每次推进项目，只做两件事是不够的：

- 不是只写代码
- 也不是只写文档

必须同时满足下面三件事：

1. 当前目标清楚
2. 当前证据清楚
3. 当前回退点清楚

只要这三件事清楚，项目就会稳；只要缺一个，后面就容易返工、跑偏或误覆盖。
