4. 最后再进入 `tea-trace-system/README.md` 和 `tea-trace-system/docs/` 查看具体说明

## 2. 项目总目标

完成一个可运行、可演示、适合作为课程项目提交的全栈系统：

- 项目名称：基于区块链的茶叶全流程溯源系统设计与实现
- 技术栈固定：`Vue 3 + Vite + Pinia + Vue Router + Element Plus + Node.js + Express + MySQL + Hardhat + Solidity + ethers + Ganache`
- 开发语言固定：全部使用 `JavaScript`
- 核心模式固定：链下存详情，链上存摘要
- 项目定位固定：优先保证可运行、可演示、结构清晰，不做过度设计

---

## 3. 固定边界与基本原则

整个项目推进过程中，以下内容不要改：

1. 不更换技术栈，不引入 TypeScript，不使用 ORM，不做微服务，不使用 Docker。
2. 先完成 MVP，再补细节，不要一开始追求“大而全”。
3. 先打通数据库和后端，再接区块链，再做前端页面。
4. 后端统一接口返回结构：`code`、`message`、`data`。
5. 所有时间字段统一格式：`YYYY-MM-DD HH:mm:ss`。
6. 所有上传文件先落地到 `backend/uploads`。
7. 所有区块链写操作由后端发起，前端不直接操作链。
8. 所有阶段都要以“能运行、能演示、能截图、能讲解”为验收标准。

## 4. 推荐执行方式

为了避免后期返工，后续每个阶段都必须按固定闭环执行，不再采用“写完再说”的推进方式。

### 4.1 固定闭环

每项重大改进都必须完整经过下面 7 步：

1. `规划`：明确阶段目标、任务拆分、范围边界、出口条件，并先更新进度总表
2. `Git 保护`：确认当前分支、工作区状态和稳定回退点，必要时新建分支、先提交基线、先打标签
3. `开发`：只做本轮范围内的改动，不顺手扩需求
4. `测试`：本地真实验证，不允许只写代码不验证
5. `确认`：确认功能、字段、路径、演示链路都符合目标
6. `文档`：同步更新总进度表、流程文档、README 或专题文档
7. `Git 收口`：检查差异并形成清晰提交；到关键里程碑时补 tag

只有七步全部完成，当前任务才允许从“进行中”变为“已完成”。

### 4.2 文档协同顺序

后续推进统一按下面顺序参考文档：

1. `AGENTS.md`
2. `茶叶溯源系统项目进度总表.md`
3. 本文档当前阶段小节
4. `tea-trace-system/README.md`
5. `tea-trace-system/docs/*.md`

### 4.3 统一状态字典

建议后续所有表格和记录统一使用下面状态：

- `未开始`
- `规划中`
- `开发中`
- `开发完成待测试`
- `测试通过待确认`
- `已确认待文档`
- `待联调`
- `已完成`
- `需修复`
- `阻塞`

### 4.4 Git 安全推进法

从现在开始，任何重大改动都要满足以下规则：

1. 先执行 `git status --short --branch`
2. `main` 只保留稳定状态
3. 新工作优先在 `codex/*` 分支推进
4. 关键节点必须有 commit，关键里程碑建议有 tag
5. 默认禁止使用 `git reset --hard` 这类破坏性回退方式
6. 文件、数据库、合约等高风险改动前必须先有可回退点

### 4.5 阶段出口标准

任一阶段结束前，至少满足下面 5 项：

1. 本阶段交付物已产出
2. 本阶段最低验证已完成
3. 当前不存在阻塞下一阶段的大问题
4. 相关文档已同步
5. Git 中已经留下稳定历史
