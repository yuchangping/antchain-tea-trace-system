## 5. 每次开始前的固定顺序

任何一次新会话、新开发任务、新文档维护，都必须按下面顺序执行：

1. 先读 `AGENTS.md`
2. 再读 `茶叶溯源系统项目进度总表.md`，确认当前阶段、当前里程碑、阻塞项、下一步
3. 再读 `茶叶溯源系统分阶段工作流程.md` 中对应阶段的目标、交付物、完成标准
4. 执行 `git status --short --branch`，确认当前分支和工作区是否安全
5. 判断这次任务是不是“重大改动”
6. 如果是重大改动，先建立 Git 保护点，再开始改动
7. 把本次任务拆成可闭环的小项，先更新到进度总表，再开始动手
8. 完成后必须走完测试、确认、文档、Git 收口四步，不能只停在“代码写完”

如果以上顺序没走完，不允许直接进入开发。

## 7. 什么叫重大改动

以下任一情况都视为重大改动，必须启用完整闭环和 Git 保护：

1. 修改数据库结构、索引、初始化数据或导入导出逻辑
2. 修改智能合约、部署脚本、链上数据结构或合约地址读取方式
3. 修改认证流程、接口返回结构、关键业务字段命名
4. 修改二维码、公开查询、批次创建、溯源记录这些主流程功能
5. 涉及两个或以上目录同时变更
6. 一次改动超过一个里程碑子任务
7. 任何可能影响演示链路、文档一致性或已有数据可用性的改动

文档大改也算重大改动，因为它会改变后续执行方式。

## 8. 每项重大改动的固定闭环

每一个重大改动，都必须严格走完下面七步，不能跳：

1. `规划`
   明确任务目标、范围、拆分项、出口条件，并更新进度总表。
2. `Git 保护`
   确认当前工作树状态，必要时新建分支、提交稳定点、打里程碑标签。
3. `开发`
   只做本次计划内的内容，不顺手扩需求。
4. `测试`
   至少完成本地可执行验证，留下命令、结果或可复现证据。
5. `确认`
   检查功能、字段、路径、演示链路是否真的符合目标，不以“理论上可以”代替确认。
6. `文档`
   同步更新进度总表、阶段流程文档、README 或专题文档。
7. `Git 收口`
   查看差异，提交清晰 commit；到达稳定里程碑时打 tag。

只有七步都完成，该任务才允许从 `进行中` 变为 `已完成`。

## 9. Git 安全规则

### 9.1 分支策略

1. `main` 只保留稳定状态
2. 所有新增重大工作优先在 `codex/*` 分支推进
3. 一个分支只做一个明确主题，不混做多条主线

### 9.2 提交策略

1. 先有稳定基线，再做新改动
2. 每完成一个可验证的小闭环就提交一次
3. commit 信息必须能说明变更目的，例如：
   `docs: define project execution entry`
   `feat: add backend auth bootstrap`
   `fix: align trace record field names`

### 9.3 标签策略

达到关键里程碑时建议打标签，例如：

- `baseline-20260418`
- `milestone-m4-backend-base`
- `milestone-m5-api-ready`
- `release-demo-v1`

### 9.4 禁止事项

以下操作默认禁止，除非你非常明确知道后果并且已经留好回退点：

1. `git reset --hard`
2. `git checkout -- <file>`
3. 强制删除文件后不做比对
4. 在未确认差异前覆盖现有文档或脚本
5. 一口气提交多阶段混合改动

### 9.5 推荐回退方式

1. 回退单个文件：`git restore --source <commit> -- <path>`
2. 回退某次提交：`git revert <commit>`
3. 临时搁置未完成工作：`git stash push -u -m "说明"`
4. 回到稳定里程碑：切换到对应 tag 或从 tag 新开修复分支

## 12. 状态字典

进度表中的状态只允许使用下面这些值：

- `未开始`
- `规划中`
- `开发中`
- `开发完成待测试`
- `测试通过待确认`
- `已确认待文档`
- `待联调`
- `已完成`
- `需修复`
- `阻塞`

状态切换规则：

1. 没有规划，不允许直接进入 `开发中`
2. 没有测试，不允许从 `开发中` 直接到 `已完成`
3. 没有文档同步，不允许从 `测试通过待确认` 直接到 `已完成`
4. 联调问题统一进入 `需修复` 或 `阻塞`
